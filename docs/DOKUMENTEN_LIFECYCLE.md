# Dokumenten-Lifecycle вҖ“ Lebenszyklus aller GeschГӨftsdokumente

## Гңbersicht

Dieses Dokument beschreibt den vollstГӨndigen Lebenszyklus aller GeschГӨftsdokumente im System вҖ“ sowohl ausgehende Dokumente (Rechnungen, Angebote etc.) als auch eingehende Dokumente (Lieferantenrechnungen, Lieferscheine etc.).

### 2.4 Datei-Handling

**Kernprinzip: Keine Datei-Duplikation.**

- E-Mail-AnhГӨnge: `LieferantDokument` referenziert `EmailAttachment` Гјber FK (`attachment_id`) вҖ“ die Datei wird **nicht** kopiert
- Manuelle Uploads: Dateien werden im `uploads/`-Verzeichnis mit UUID-basiertem Namen gespeichert (`gespeicherterDateiname`)
- Effektiver Dateiname: `getEffektiverDateiname()` priorisiert Attachment, dann Fallback auf `originalDateiname`

## 3. Projekt-Zuordnung und Kostenverteilung

### 3.1 LieferantDokumentProjektAnteil

Jedes Lieferanten-Dokument kann **mehreren Projekten oder Kostenstellen** prozentual zugeordnet werden:

```
Lieferantenrechnung (Brutto: 10.000 вӮ¬)
   в”Ӯ
   в”ңв”Җв”Җ Projekt A: 60% вҶ’ 6.000 вӮ¬
   в”ңв”Җв”Җ Projekt B: 30% вҶ’ 3.000 вӮ¬
   в””в”Җв”Җ Kostenstelle "Lager": 10% вҶ’ 1.000 вӮ¬
```

### 3.2 Zuordnungsmethoden

| Methode | Feld | Beschreibung |
|---|---|---|
| **Prozentual** | `prozent` | Prozentsatz (0вҖ“100) vom Gesamtbetrag |
| **Absolutbetrag** | `absoluterBetrag` | Fester Betrag (Гјberschreibt Prozent) |

Die Berechnung erfolgt in `berechneAnteil()`:
- Wenn `absoluterBetrag` gesetzt вҶ’ direkt verwenden
- Sonst: `gesamtBetrag Г— prozent / 100`

### 3.3 Kostenstreckung

FГјr periodische Kosten (z.B. Zertifizierung alle 4 Jahre) unterstГјtzt die Entity eine **Kostenstreckung**:

| Feld | Beschreibung |
|---|---|
| `streckungJahre` | Гңber wie viele Jahre verteilen (Default: 1) |
| `streckungStartJahr` | Ab welchem Jahr gilt die Streckung |

**Beispiel:** Zertifizierung 4.000 вӮ¬ alle 4 Jahre:
- `streckungJahre = 4`, `streckungStartJahr = 2026`
- `getJahresanteil()` вҶ’ 1.000 вӮ¬ pro Jahr
- `isStreckungAktivFuerJahr(2027)` вҶ’ `true`
- `isStreckungAktivFuerJahr(2030)` вҶ’ `false`

### 3.4 Lagerbestellung-Flag

Dokumente kГ¶nnen als `lagerbestellung = true` markiert werden. In diesem Fall ist **keine Projekt-Zuordnung erforderlich**.

## 4. VerknГјpfung zwischen Dokumenten

### 4.1 Ausgangs-Dokumente: Lineare Kette

Ausgangs-GeschГӨftsdokumente verwenden eine **lineare VorgГӨnger/Nachfolger-Kette** Гјber FK:

```java
@ManyToOne vorgaenger        // вҶ’ VorgГӨnger-Dokument
@OneToMany nachfolger        // вҶҗ Alle Nachfolger
```

### 4.2 Lieferanten-Dokumente: M:M-VerknГјpfung

Lieferanten-Dokumente verwenden eine **Many-to-Many-VerknГјpfung** fГјr flexiblere Ketten:

```java
@ManyToMany verknuepfteDokumente    // вҶ’ VerknГјpfte Dokumente
@ManyToMany verknuepftVon           // вҶҗ Dokumente, die dieses verknГјpft haben
```

**Join-Tabelle:** `lieferant_dokument_verknuepfung`

Beispiel:
```
Lieferanten-Rechnung вҶҗв”Җв”Җв–ә Lieferschein вҶҗв”Җв”Җв–ә AuftragsbestГӨtigung вҶҗв”Җв”Җв–ә Angebot
```

### 4.3 Eingangs-Dokumente: Geschaeftsdokument-Kette

Die Entity `Geschaeftsdokument` (im Projekt-Kontext) nutzt ebenfalls eine lineare FK-Kette:

```java
@ManyToOne vorgaengerDokument
@OneToMany nachfolgerDokumente
```
